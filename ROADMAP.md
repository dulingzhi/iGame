# iGame 项目路线图（ROADMAP）

> **目标**：构建一个**数据驱动（ECS）**的跨平台游戏运行时（Desktop + Web/wasm），并提供类似《魔兽争霸III世界编辑器》的 **UGC 编辑器**，让玩家自行制作地图/玩法并发布试玩。  
> 运行时与编辑器共享同一套数据格式、资产规范与校验体系。  
> 技术栈：**Rust / Bevy（ECS）**，支持 Desktop（Windows/macOS/Linux）与 Web（wasm）。

## 1. 产品范围与非目标

### 1.1 核心范围（必须交付）

| 层次 | 说明 |
|------|------|
| **Runtime（运行时）** | 加载地图包 → 实例化世界（ECS）→ 运行规则与触发器 → 桌面 + Web 均可运行 |
| **Editor（编辑器，桌面优先）** | 场景编辑、资源管理、Inspector、触发器节点图、Play-In-Editor（PIE） |
| **MapPackage（地图包体系）** | 地图包格式规范、依赖管理、版本迁移、完整校验与错误提示 |
| **UGC 平台基础** | 内容索引、打包发布、Web 试玩入口、基础安全与沙箱（脚本路线时必须） |
| **可扩展性** | 插件/模块化架构（引擎侧），后续支持脚本或更安全的扩展机制 |

### 1.2 非目标（第一阶段明确不做）

- 复杂 AAA 渲染特性（全局光照、电影级后处理）——优先级低，后期可选
- 全功能 Web 编辑器——先做 Web 试玩/预览；Web 编辑器放后期
- "允许玩家写 Rust 并编译"——跨平台与安全/分发成本过高，不作为主路线
- 移动端（iOS/Android）——本期不做，架构上不阻断

## 2. 总体架构建议

### 2.1 分层结构

```
┌──────────────────────────────────────────┐
│             UGC Platform（内容分发层）      │
│  内容索引 / 打包发布 / Web 试玩 / 签名      │
├──────────────────────────────────────────┤
│             Editor（编辑器层）              │
│  Viewport / Inspector / Gizmo            │
│  触发器图编辑器 / 资源浏览器 / PIE           │
├──────────────────────────────────────────┤
│             Runtime（运行时层）             │
│  ECS（Components/Resources/Events）       │
│  Schedules/Systems / 触发器解释器           │
│  Asset Server / MapPackage Loader         │
└──────────────────────────────────────────┘
         ↕ 共享：数据格式 / 序列化 / 校验
```

### 2.2 建议 Workspace 结构

```
iGame/
├── crates/
│   ├── runtime/      # 游戏运行时（Bevy App、ECS、规则、触发器解释器）
│   ├── editor/       # 编辑器（Bevy App + UI、场景视图、触发器图 UI）
│   ├── shared/       # 共享数据结构、序列化、校验、版本迁移
│   └── ugc/          # 包管理、依赖解析、内容索引
├── examples/         # 最小可运行示例
├── docs/             # 设计文档
├── ROADMAP.md
├── README.md
└── Cargo.toml        # workspace 根
```

### 2.3 地图包（MapPackage）格式规范 v1

地图包以**目录或 zip 压缩包**为载体，结构如下：

```
my_map/
├── manifest.toml         # 元信息（名称/作者/版本/引擎最小版本/依赖/入口场景）
├── scene/
│   └── main.json         # 场景实体与组件（ECS 序列化，含版本字段）
├── rules/
│   ├── units.toml        # 单位/技能/物品数值表
│   └── items.toml
├── triggers/
│   └── main_triggers.json  # 触发器图（节点与连线）
├── assets/
│   ├── textures/
│   ├── models/
│   └── audio/
├── localization/
│   ├── zh-CN.toml
│   └── en.toml
└── checksums/            # （可选）文件哈希与签名
```

**manifest.toml 关键字段示例：**

```toml
[map]
id = "my_awesome_map"
name = "我的地图"
version = "1.0.0"
author = "作者名"
engine_min_version = "0.1.0"
entry_scene = "scene/main.json"
preview_image = "assets/preview.png"

[dependencies]
common_assets = "^1.0"
```

**版本迁移与校验原则：**

- 所有序列化文件均包含 `schema_version` 字段
- 提供 `migrator` 接口，支持逐版本升级（v1 → v1.1 → v2）
- 校验器覆盖：结构校验、资源引用校验、循环依赖检测、平台兼容性校验（Web 限制）
- 校验失败需给出清晰错误信息（文件路径、字段名、失败原因）

### M0 — 仓库初始化与工程基建

| 项 | 内容 |
|----|------|
| **目标** | 可持续开发的工程骨架与规范落地 |
| **建议周期** | 1 周 |

**范围：**
- 初始化 Cargo workspace，建立 `crates/runtime`、`crates/editor`、`crates/shared`、`crates/ugc` 骨架
- 配置 CI：`cargo fmt` / `cargo clippy` / `cargo test` / `cargo build --target wasm32-unknown-unknown`
- 基础文档：README + ROADMAP + CONTRIBUTING（简版）
- 确定关键依赖版本（Bevy、序列化库、UI 库）
- 约定代码风格与提交规范

**DoD（验收标准）：**
- [ ] 本地一键 `cargo test` 全部通过
- [ ] CI 配置完成并在 GitHub Actions 绿色
- [ ] README 可运行一个最小 hello-world demo
- [ ] ROADMAP.md、CONTRIBUTING.md 存在且可读

**关键风险/依赖：**
- Bevy 版本选择影响 wasm 兼容性，需要早期锁定
- wasm feature flags 需要规划（线程、文件系统 API 限制）

### M1 — Runtime MVP：场景加载 + 渲染 + 输入相机

| 项 | 内容 |
|----|------|
| **目标** | 能在桌面和 Web 加载一个"地图"并进行基础互动 |
| **建议周期** | 2 ~ 4 周 |

**范围：**
- 基础 App 初始化、日志系统、状态机（`Loading` / `Playing` / `Paused`）
- RTS 相机控制（平移/缩放/旋转）
- 键鼠输入系统（Web 兼容）
- 基础 2D/3D 渲染（放置实体：地面、单位占位、UI）
- 基础资源加载（纹理/模型/字体）

**DoD：**
- [ ] Desktop：加载示例地图，目标 60fps（开发机）
- [ ] Web：同一示例地图可在浏览器运行（帧率允许下降，但可操作）
- [ ] 相机可正常操作，点击拾取有反馈

**关键风险/依赖：**
- Web 平台限制（线程模型、文件系统、音频、网络）需在此阶段踩坑并记录
- wasm bundle 体积需要评估压缩策略

### M2 — MapPackage v1：序列化模型与校验

| 项 | 内容 |
|----|------|
| **目标** | 地图成为可导入/导出的"内容包"，运行时可加载并实例化 ECS 世界 |
| **建议周期** | 2 ~ 4 周 |

**范围：**
- `manifest.toml` 解析
- `scene` 序列化：Entity/Component 的可演进格式（含 `schema_version`）
- 资源引用：相对路径 + UUID / 逻辑 ID 双轨
- 校验器 v1：缺资源、循环依赖、非法字段、版本不兼容
- 版本迁移框架：能编写和注册 `migrator`，提供测试用例模板

**DoD：**
- [ ] 运行时可加载 `MapPackage`，生成对应 ECS 实体
- [ ] 校验失败输出清晰错误（文件、字段、原因）
- [ ] 至少 2 个迁移测试用例通过

**关键风险/依赖：**
- 序列化格式一旦对外发布，迁移成本会放大 → 需要提前设计"版本迁移"机制
- ECS 组件结构变化时需保证迁移脚本可用

### M3 — Editor MVP：场景视图 + Inspector + 保存/加载

| 项 | 内容 |
|----|------|
| **目标** | 具备 War3 编辑器"放东西、改属性、保存"的最小闭环 |
| **建议周期** | 3 ~ 6 周 |

**范围：**
- Viewport：相机/网格背景/选中高亮
- 基础 Gizmo：移动/旋转/缩放（至少移动）
- Hierarchy 面板：实体列表、分组、搜索
- Inspector 面板：编辑组件字段（位置/旋转/缩放/资源引用/自定义字段）
- 资源浏览器：导入资产、预览、引用
- Save/Load：输出/读入 `MapPackage v1`
- PIE（Play-In-Editor）：一键在编辑器内启动运行时

**DoD：**
- [ ] 作者可新建地图、放置实体、修改属性、保存
- [ ] 运行时加载保存的地图并一致呈现
- [ ] PIE 可在 5 秒内进入运行状态

**关键风险/依赖：**
- UI 框架选型（egui / bevy_ui / 其他）影响编辑器效率与可维护性，需尽早决策

### M4 — 触发器系统 MVP：可视化节点图 + 解释执行

| 项 | 内容 |
|----|------|
| **目标** | 实现 War3 核心：事件-条件-动作（ECA）触发器，作者无需写代码即可做玩法逻辑 |
| **建议周期** | 3 ~ 6 周 |

**范围：**
- Trigger Graph 数据结构：节点（Node）、端口（Port）、连线（Edge）、变量（Variable）、常量（Const）
- 运行时解释器：按事件触发执行，支持条件分支与变量读写
- 调试支持：执行日志、节点 ID 定位、错误信息
- 编辑器：节点图 UI（拖拽连线、节点搜索、参数内联编辑）
- 首批 MVP 节点（详见第 5 节）

**DoD：**
- [ ] 触发器能实现：刷怪波次 / 胜利条件 / 对话提示 3 个玩法示例
- [ ] 触发器错误能定位到具体节点与失败原因
- [ ] 触发器数据可随 MapPackage 保存/加载

**关键风险/依赖：**
- 节点系统一旦公开给作者，需要版本管理与向后兼容策略
- "Wait/Delay" 节点的实现方式需避免阻塞 ECS 调度

### M5 — Web 试玩与内容分发 MVP

| 项 | 内容 |
|----|------|
| **目标** | 用户能在 Web 上浏览并试玩地图；为后续 UGC 社区铺路 |
| **建议周期** | 2 ~ 5 周 |

**范围：**
- Web 运行时打包与资产加载策略（bundle / 远程拉取）

**DoD：**
- [ ] Web 页面可列出地图 → 点击进入 → 加载完成 → 成功游玩
- [ ] 加载失败有友好提示，不出现白屏或无响应

**关键风险/依赖：**
- 资产体积控制与加载速度（首屏等待时间目标 < 10s）
- CORS / 内容托管策略需要确定

### M6 — 资产管线增强 + 依赖与共享资源包

| 项 | 内容 |
|----|------|
| **目标** | UGC 扩展到"模块化内容"：地图可依赖公共资源包，版本可控 |
| **建议周期** | 3 ~ 6 周 |

**范围：**
- 资源包（Asset Pack）概念：与地图包分离，独立版本
- 依赖解析与冲突处理（SemVer 版本范围、锁文件）
- 校验器增强：依赖缺失 / 版本不兼容提示
- 热重载（桌面）：资产更改后自动重载，提升编辑效率
- 公共资源包发布流程（初版）

**DoD：**
- [ ] 地图可引用共享资源包并正确打包/发布/加载
- [ ] 依赖冲突有清晰错误提示
- [ ] 桌面热重载基本可用

**关键风险/依赖：**
- 依赖版本策略设计复杂，需尽早约束（SemVer、锁文件格式）

### M7 — 扩展机制 v1：插件/脚本路线与沙箱

| 项 | 内容 |
|----|------|
| **目标** | 允许高级作者做更复杂逻辑，同时保证跨平台运行与安全 |
| **建议周期** | 4 ~ 8 周（可与其他里程碑并行探索） |

**范围（路线二选一或组合）：**
- **路线 A：嵌入脚本（Lua / Rhai）**
  - API 白名单、资源访问限制、执行超时
  - 编辑器支持：脚本资源管理、错误高亮
- **路线 B：WASM 脚本（更强隔离）**
  - 工具链、性能、调试体验评估报告

**DoD：**
- [ ] 提供"自定义规则模块"示例（如自定义技能效果）
- [ ] 沙箱覆盖：API 白名单 + 资源访问限制 + 执行超时均有测试

**关键风险/依赖：**
- 沙箱与权限模型是 UGC 平台核心安全风险（防恶意代码/资源滥用/作弊）

### M8 — 联机 / 回放（可选，取决于目标游戏类型）

| 项 | 内容 |
|----|------|
| **目标** | 支持多人游戏或回放，提升作品上限 |
| **建议周期** | 6 ~ 12 周 |

**范围：**
- 回放：输入记录或事件记录（建议先做事件回放，再追求 determinism）
- 联机：权威服务器 / 锁步 / 预测回滚（根据玩法类型选择）
- Web 端网络限制评估（WebSocket / WebTransport）

**DoD：**
- [ ] 1v1 或简单 coop 原型可跑通一局
- [ ] 回放文件可重播并与原局面一致

**关键风险/依赖：**
- determinism、浮点一致性、物理一致性会显著增加复杂度
- Web 网络 API 限制（仅 WebSocket / WebTransport，无原始 UDP）

- [ ] 相机：RTS 相机（平移/缩放/旋转）

## 5. 触发器系统 MVP 节点清单

> 建议 M4 里程碑首批实现以下 20 个节点，覆盖基础 RTS/RPG/塔防玩法所需的最小集合。

### 5.1 事件节点（Events）

| # | 节点名称 | 说明 |
|---|----------|------|
| E1 | **Game Start**（游戏开始） | 游戏/地图加载完成后触发一次 |
| E2 | **Timer Every N Seconds**（周期计时器） | 每隔 N 秒触发一次（N 可配置） |
| E3 | **Unit Created**（单位创建） | 指定单位（或任意单位）被生成时触发 |
| E4 | **Unit Died**（单位死亡） | 指定单位（或任意单位）死亡时触发 |
| E5 | **Unit Enter Region**（单位进入区域） | 指定单位进入指定区域时触发 |
| E6 | **Unit Leave Region**（单位离开区域） | 指定单位离开指定区域时触发 |
| E7 | **Custom Event Fired**（自定义事件） | 监听由 Fire Custom Event 动作触发的自定义事件 |

### 5.2 条件节点（Conditions）

| # | 节点名称 | 说明 |
|---|----------|------|
| C1 | **Compare Int / Float**（数值比较） | 比较两个数值（`==` / `!=` / `>` / `>=` / `<` / `<=`） |
| C2 | **And / Or / Not**（逻辑组合） | 多个条件的逻辑组合节点 |
| C3 | **Unit Has Tag**（单位标签判断） | 判断单位是否拥有指定 Tag |
| C4 | **Faction Is**（阵营判断） | 判断单位阵营 |
| C5 | **Random Chance**（随机概率） | 以百分比概率返回 true（如 30% 触发） |
| C6 | **Is In Region**（区域判断） | 判断实体当前是否在指定区域内 |

### 5.3 动作节点（Actions）

| # | 节点名称 | 说明 |
|---|----------|------|
| A1 | **Spawn Unit**（生成单位） | 在指定点/区域随机点生成一个单位 |
| A2 | **Destroy Entity**（销毁实体） | 销毁指定实体（单位/特效/物件等） |
| A3 | **Set Variable**（变量赋值） | 给全局/地图变量赋值 |
| A4 | **Add / Remove Component**（组件增删） | 动态添加或移除 ECS 组件（用于状态/Buff） |
| A5 | **Play Sound**（播放音效） | 在指定位置或全局播放音效 |
| A6 | **Play VFX**（播放特效） | 在指定位置/实体上播放视觉特效 |
| A7 | **Show Message / UI Toast**（UI 提示） | 显示屏幕提示文字（可设持续时间） |
| A8 | **Set Victory / Defeat**（胜利/失败） | 触发指定玩家的胜利或失败结算 |
| A9 | **Fire Custom Event**（触发自定义事件） | 向系统广播一个自定义事件（供 E7 监听） |
| A10 | **Wait / Delay**（延迟执行） | 延迟 N 秒后继续执行后续动作（注意：需异步实现，避免阻塞调度） |

> **注意**：Wait/Delay 节点需要使用 Bevy 的异步/协程机制或状态机方案实现，避免阻塞 ECS System。

## 7. 垂直切片验收样例地图

> 建议尽早定义并维护一张"垂直切片地图（Vertical Slice Map）"作为贯穿全周期的验收基准，每个里程碑均需保证此地图可运行。

### 7.1 样例地图规格（"防守反击"）

| 要素 | 描述 |
|------|------|
| **地形** | 一张 32×32 格地图，含路径、障碍物、基地区域 |
| **单位类型** | 1 种敌方单位（拥有移动、攻击、死亡动画占位） |
| **刷怪触发器** | 每 30 秒生成一波敌方单位（使用 E2 + A1） |
| **计数器** | 击杀敌方单位计数（使用 E4 + A3 Set Variable） |
| **胜利条件** | 击杀数达到 20 触发胜利（使用 C1 + A8 Victory） |
| **失败条件** | 基地区域被占领（使用 E5 Unit Enter Region + A8 Defeat） |
| **UI 提示** | 每波刷怪时显示提示文字（使用 A7） |
| **音效** | 胜利/失败时播放音效（使用 A5） |

### 7.2 各里程碑对样例地图的要求

| 里程碑 | 对样例地图的最低要求 |
|--------|----------------------|
| M1 | 地图可加载，相机可操作，单位占位可见 |
| M2 | 地图包可序列化/反序列化，实体正确生成 |
| M3 | 编辑器可打开并修改地图，PIE 可运行 |
| M4 | 触发器全部运行，胜利/失败/刷怪逻辑正常 |
| M5 | Web 可加载并试玩完整一局 |
| M9 | 地图在 Desktop + Web 稳定运行 10 局以上 |
| M10 | 地图可按文档从零复现，作为官方示例发布 |
